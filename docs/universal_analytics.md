## Executive Summary

### Current State
Our charting system has **6+ divergent API endpoints**, **4 different data fetch patterns**, and **split server/client transformation logic**. The main orchestrator component ([analytics-chart.tsx](../components/charts/analytics-chart.tsx)) is 780 lines with deep conditional branching.

### Goal
Create a **unified charting workflow** with:
- ‚úÖ Single API gateway for all chart types
- ‚úÖ 100% server-side data transformation
- ‚úÖ Pluggable chart type system (registry pattern)
- ‚úÖ Simplified components (<200 lines)
- ‚úÖ Type-safe configurations with Zod
- ‚úÖ Unified Redis caching strategy

### Benefits
- üöÄ **30-50% faster dashboard loads** (batched queries)
- üîß **70% reduction** in component complexity
- üéØ **Easy extensibility** for new chart types
- üõ°Ô∏è **Type safety** with compile-time validation
- ‚ö° **Consistent caching** across all chart types

## Problems Identified

### CRITICAL ISSUES

#### 1. ‚ùå API Endpoint Fragmentation
- **6+ different endpoints** for fetching chart data
- No consistent request/response format
- Difficult to add new chart types (need new endpoint + client logic)
- Code duplication across endpoints

#### 2. ‚ùå Transformation Logic Split
- **Inconsistent patterns:**
  - ‚úÖ Standard charts: Server-side transformation (GOOD)
  - ‚ùå Number charts: Client-side aggregation (BAD)
  - ‚ùå Dual-axis: Client-side transformation (BAD)
  - ‚ö†Ô∏è Progress bars: Server + client (CONFUSING)
  - ‚ùå Tables: No transformation (raw data)

#### 3. ‚ùå AnalyticsChart Component Bloat
- **780 lines** with deep conditional nesting
- **4 completely different** data fetch patterns
- Violates **Single Responsibility Principle**
- Hard to test, hard to maintain

#### 4. ‚ùå Inconsistent Data Flow
```
Standard Charts:  API ‚Üí Server Transform ‚Üí Client Render
Table Charts:     Different API ‚Üí Raw Data ‚Üí Client Render
Number Charts:    Measures API ‚Üí Client Aggregate ‚Üí Client Render
Dual-Axis:        Dual API ‚Üí Client Transform ‚Üí Client Render
```

#### 5. ‚ùå Type Safety Issues
- Chart configs stored as JSONB (no compile-time validation)
- Runtime type casting throughout client code
- No schema validation on save

#### 6. ‚ùå Caching Inconsistency
- Server transformation results **not cached**
- Client-side caching via React state only (lost on unmount)
- No unified caching strategy
- Each endpoint has different caching approach

## Proposed Architecture

### Design Principles

1. ‚úÖ **Single API Gateway** - One endpoint for ALL chart data
2. ‚úÖ **Server-Side Transformation** - All transformation on server
3. ‚úÖ **Type-Safe Configs** - Zod schemas replace JSONB where possible
4. ‚úÖ **Component Simplification** - Thin rendering components only
5. ‚úÖ **Pluggable Chart Types** - Registry pattern for extensibility
6. ‚úÖ **Unified Caching** - Redis-backed with consistent TTL

interface CacheConfig {

// Cache Key Generation (deterministic from chart config)
function generateCacheKey(config: ChartConfig): string {
  const configHash = hashObject({
    chartType: config.chartType,
    dataSourceId: config.dataSourceId,
    filters: config.filters,
    groupBy: config.groupBy,
    // ... other relevant fields
  });
  return `chart:${config.chartType}:${config.dataSourceId}:${configHash}`;

// Cache Implementation
class ChartDataCache {
  async get(key: string): Promise<UnifiedChartDataResponse | null> {
    return redisClient.get(key);
  }

  async set(
    key: string,
    data: UnifiedChartDataResponse,
    ttl: number = 300
  ): Promise<void> {
    await redisClient.setex(key, ttl, JSON.stringify(data));
  }

  async invalidate(pattern: string): Promise<void> {
    // Invalidate by pattern (e.g., "chart:bar:*")
    const keys = await redisClient.keys(pattern);
    await redisClient.del(...keys);
  }
}

## Risk Mitigation

### 1. Backward Compatibility

**Risk:** Breaking existing integrations or dashboards

**Mitigation:**
- ‚úÖ Keep old endpoints during migration (mark as deprecated)
- ‚úÖ Feature flag for universal endpoint (gradual rollout)
- ‚úÖ Gradual migration per chart type (can rollback individual types)
- ‚úÖ Maintain API contract compatibility
- ‚úÖ 2-week deprecation notice before removing old endpoints

### 2. Performance Regression

**Risk:** New system slower than current implementation

**Mitigation:**
- ‚úÖ Performance benchmarks before/after each phase
- ‚úÖ Load testing with production-like data volumes
- ‚úÖ Monitoring and alerting for P95 latency
- ‚úÖ Redis caching to offset any query overhead
- ‚úÖ Database query optimization (indexes, query plans)

### 3. Data Inconsistency

**Risk:** Server/client transformation differences cause visual bugs

**Mitigation:**
- ‚úÖ Comprehensive visual regression tests
- ‚úÖ Side-by-side comparison tool (old vs new)
- ‚úÖ Staged rollout (canary deployment)
- ‚úÖ Detailed logging of transformation inputs/outputs
- ‚úÖ E2E tests covering all chart types

### 4. Migration Complexity

**Risk:** Schema migration fails or corrupts data

**Mitigation:**
- ‚úÖ Full database backup before any migration
- ‚úÖ Dry-run mode for all migration scripts
- ‚úÖ Validation reports before applying changes
- ‚úÖ Rollback scripts for each migration
- ‚úÖ Test migrations on copy of production data

### 5. Cache Invalidation Bugs

**Risk:** Stale data shown to users after updates

**Mitigation:**
- ‚úÖ Conservative TTL (5 minutes)
- ‚úÖ Manual refresh button always bypasses cache
- ‚úÖ Comprehensive invalidation on any config change
- ‚úÖ Cache key versioning (increment version on schema change)
- ‚úÖ Monitoring for cache-related issues

### 6. Testing Gaps

**Risk:** Regressions not caught until production

**Mitigation:**
- ‚úÖ Unit tests for all new code (>85% coverage)
- ‚úÖ Integration tests for all chart types
- ‚úÖ E2E tests for critical user journeys
- ‚úÖ Visual regression tests (screenshot comparison)
- ‚úÖ Staging environment with production data clone
- ‚úÖ Beta testing with select users before full rollout

### 7. Rollback Strategy

**Risk:** Need to quickly revert changes if critical issues found

**Mitigation:**
- ‚úÖ Feature flags for easy disable
- ‚úÖ Old endpoints remain functional (parallel systems)
- ‚úÖ Database migrations are reversible
- ‚úÖ Deployment strategy allows instant rollback
- ‚úÖ Documented rollback procedures
- ‚úÖ On-call rotation during rollout period
